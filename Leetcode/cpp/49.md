# Group Anagrams

## 1. Problem Description

Given an array of strings `strs`, group the **anagrams** together.

Two strings are anagrams if they contain the same characters with the same frequency but possibly in different orders.

Example

Input

["eat","tea","tan","ate","nat","bat"]

Output

[
 ["eat","tea","ate"],
 ["tan","nat"],
 ["bat"]
]

---

## 2. Key Observation

Two strings are anagrams **iff their character frequency distribution is identical**.

Example

eat
tea
ate

Character frequency

a = 1
e = 1
t = 1

Frequency vector

[1,0,0,0,1,0,...,1,...]

Therefore we can build a **canonical representation** for each string.

---

## 3. Approach 1 — Frequency Encoding (Recommended)

Use a fixed-size array to record character frequency.

array<uint8_t,26>

Index mapping

0 -> 'a'
1 -> 'b'
...
25 -> 'z'

Example

eat -> [1,0,0,0,1,0,...,1,...]  
tea -> [1,0,0,0,1,0,...,1,...]

Both produce the same key.

We use this key in a hash table.

---

## 4. C++ Implementation

    class Solution {
    public:

        struct ArrayHash {
            size_t operator()(const array<uint8_t,26>& arr) const noexcept {

                size_t hash = 0;

                for(uint8_t v : arr)
                    hash = hash * 31 + v;

                return hash;
            }
        };

        vector<vector<string>> groupAnagrams(vector<string>& strs) {

            unordered_map<array<uint8_t,26>, vector<string>, ArrayHash> mp;
            mp.reserve(strs.size());

            for(const string& s : strs) {

                array<uint8_t,26> freq{};

                for(char c : s)
                    freq[c-'a']++;

                mp[freq].push_back(s);
            }

            vector<vector<string>> res;
            res.reserve(mp.size());

            for(auto& [k,v] : mp)
                res.push_back(move(v));

            return res;
        }
    };

---

## 5. Time Complexity

Let

n = number of strings  
k = average string length

Time Complexity

O(n * k)

Reason

frequency counting per string = O(k)  
hash computation = O(26) ≈ O(1)

---

## 6. Space Complexity

Space Complexity

O(n * k)

Because all strings must be stored in the output.

---

## 7. Why Using uint8_t

Using

array<int,26>

Key size

26 * 4 bytes = 104 bytes

Using

array<uint8_t,26>

Key size

26 bytes

Benefits

- smaller key
- faster hashing
- faster comparison
- better cache locality

---

## 8. Hash Function Design

The hash function converts a key into an integer.

Example

hash = hash * 31 + value

This is called **Polynomial Rolling Hash**.

Where

31 = prime multiplier.

---

## 9. Important Concept — Hash Collision

A very important concept:

Hash function **can have collisions**.

Different keys may produce the same hash value.

Example

key1 -> hash = 12345  
key2 -> hash = 12345

This does NOT break correctness.

Because unordered_map performs an additional **key comparison**.

---

## 10. unordered_map Lookup Process

The lookup process

1. compute hash(key)
2. locate the bucket
3. iterate entries inside bucket
4. use cmp function to check equality

Flow

key  
↓  
hash function  
↓  
bucket  
↓  
cmp function  

Therefore

hash -> locate bucket  
cmp -> determine equality

---

## 11. Four Core Components of unordered_map

Full template

    unordered_map<
        Key,
        Value,
        HashFunction,
        KeyEqual
    >

Example

    unordered_map<string,int>

Equivalent to

    unordered_map<
        string,
        int,
        std::hash<string>,
        std::equal_to<string>
    >

1. Key

lookup key

Examples

string  
array<int,26>

2. Value

stored data

Example

vector<string>

3. Hash Function

convert key → hash value

Examples

std::hash<string>  
custom hash

4. Key Equality Function (cmp)

determine if two keys are equal

Default

std::equal_to<Key>

---

## 12. Approach 2 — Sort String

Another common approach is sorting the string.

Example

eat -> aet  
tea -> aet  
ate -> aet  

Implementation

    unordered_map<string, vector<string>> mp;

    for(string s : strs) {

        string key = s;

        sort(key.begin(), key.end());

        mp[key].push_back(s);
    }

---

## 13. Complexity Comparison

Frequency Encoding

Time: O(nk)  
Key Size: 26 bytes  

Sort String

Time: O(n k log k)  
Key Size: k bytes  

---

## 14. When to Choose Each Approach

If

k is small (<20)

Sorting is usually fine.

If

k is large

Frequency encoding is better.

---

## 15. Summary

The core of Group Anagrams is building a **canonical representation**.

Two common approaches

1. Sort string
2. Character frequency encoding

Engineering-wise

Frequency Encoding + Hash Table

is often preferred because

- fixed key size
- stable hashing
- O(nk) time complexity
